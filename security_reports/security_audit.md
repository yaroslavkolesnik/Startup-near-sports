# Мобильное приложение Sports Meetup: Отчет об аудите безопасности

**Дата:** 2026-06-13
**Инструмент:** AI Security Tester (OWASP Mobile & API Top 10)

В ходе автоматизированного анализа исходного кода MVP проекта (Frontend: React Native, Backend: Django DRF) были выявлены следующие архитектурные решения, уязвимости и пробелы в безопасности.

---

## 🟢 Положительные моменты (Что сделано хорошо)
* **Безопасное хранение токенов:** Frontend использует `expo-secure-store` для хранения JWT (Access и Refresh токенов), что защищает их от извлечения злоумышленниками. `AsyncStorage` для критичных данных не используется.
* **Защита от SQL-инъекций:** Backend использует исключительно Django ORM, что автоматически параметризует SQL-запросы и нейтрализует угрозы внедрения кода.
* **Валидация паролей:** В `settings.py` подключены стандартные валидаторы паролей (`MinimumLengthValidator`, `NumericPasswordValidator` и т.д.), и они корректно вызываются в сериализаторе регистрации.
* **Изоляция сессий (Throttling):** В настройках DRF настроен Rate Limiting (5 запросов в минуту для анонимов, 120 для пользователей), что защищает от брутфорса и DDoS-атак базового уровня.

## 🔴 Уязвимости и критические пробелы

> [!WARNING]
> Следующие уязвимости необходимо устранить до релиза в production (App Store / Google Play).

### 1. Передача данных в открытом виде (HTTP)
**Компонент:** Frontend (`src/api/auth.js`, `src/api/pitches.js`, и т.д.)
**Описание:** Базовый URL API жестко задан как `http://192.168.0.67:8000/api`. Использование протокола HTTP означает, что все данные, включая пароли при регистрации/логине и JWT-токены, передаются в открытом виде и могут быть перехвачены через атаку "Man-in-the-Middle" (MitM).
**Требуемое улучшение:** 
- Перевести production сервер на HTTPS (с настройкой SSL/TLS сертификата).
- Вынести конфигурацию URL в `.env` файлы (например, с использованием `react-native-dotenv`), чтобы локально использовать HTTP, а в проде строго HTTPS.

### 2. Слишком открытая политика CORS (Cross-Origin Resource Sharing)
**Компонент:** Backend (`config/settings.py`)
**Описание:** Параметр `CORS_ALLOW_ALL_ORIGINS = True` разрешает запросы к вашему API с любого доменного имени. Хотя мобильные приложения игнорируют CORS, если API станет публичным или уязвимым для веб-атак (CSRF с других сайтов), это создает огромную брешь.
**Требуемое улучшение:** 
- Установить `CORS_ALLOW_ALL_ORIGINS = False`.
- Настроить `CORS_ALLOWED_ORIGINS` только для доверенных доменов (если планируется веб-версия админки).

### 3. Небезопасные настройки хостов (ALLOWED_HOSTS)
**Компонент:** Backend (`config/settings.py`)
**Описание:** `ALLOWED_HOSTS` по умолчанию падает до `*` (`os.environ.get('ALLOWED_HOSTS', '*').split(',')`). В продакшене это делает сервер уязвимым для атак с подменой заголовка HTTP Host (Host Header Injection), что может вести к спуфингу сброса паролей или отравлению кэша.
**Требуемое улучшение:** В production окружении переменная `ALLOWED_HOSTS` должна строго содержать только IP-адрес сервера и его доменное имя.

### 4. Риск использования дефолтного SECRET_KEY
**Компонент:** Backend (`config/settings.py`)
**Описание:** Ключ `SECRET_KEY` падает на `'django-insecure-default'`, если переменная окружения не задана. Если сервер случайно развернется без правильно настроенного `.env`, злоумышленники смогут подделывать сессии и токены.
**Требуемое улучшение:** Оборачивать получение ключа в продакшене в блок с генерацией ошибки, если ключ отсутствует, либо использовать пакет вроде `django-environ`.

## 🟡 Архитектурные рекомендации (Улучшения)

> [!TIP]
> Рекомендации для повышения уровня зрелости защиты (Defense in Depth).

1. **Сертификатный пиннинг (SSL Pinning):** Для защиты от продвинутых MitM атак на зараженных устройствах стоит рассмотреть внедрение SSL Pinning на фронтенде (например, через библиотеку `react-native-ssl-pinning`).
2. **Проверка Deep Links:** Если в приложении будут внедрены Deep Links (например, для шаринга матчей), необходимо строго валидировать входящие параметры URL перед их парсингом и использованием в приложении (для предотвращения XSS/инъекций на уровне JS).
3. **Ротация токенов (Refresh Token Reuse):** В текущей реализации токенов (`SIMPLE_JWT`) время жизни Refresh-токена 7 дней. Стоит добавить проверку `ROTATE_REFRESH_TOKENS = True` в настройках DRF Simple JWT, чтобы при каждом обновлении access-токена выдавался новый refresh-токен (с инвалидацией старого), что минимизирует риски кражи сессии.
